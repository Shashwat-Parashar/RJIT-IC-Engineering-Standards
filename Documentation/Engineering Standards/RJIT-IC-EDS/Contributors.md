# Contributing to RJIT Innovation Club Engineering Standards

> Guidelines for contributing to the engineering documentation framework of the RJIT Innovation Club.

---

## Purpose

The purpose of this document is to establish a consistent contribution workflow for the RJIT Innovation Club Engineering Standards repository.

These guidelines help ensure that documentation remains accurate, professional, well-organized, and maintainable as the repository grows.


## Who Can Contribute

Contributions are welcome from:

- Members of the RJIT Innovation Club
- Faculty mentors
- Student project teams
- Repository maintainers
- Future contributors approved by the maintainers


## Types of Contributions

Contributors may assist by:

- Developing engineering documentation standards
- Improving existing documentation
- Creating engineering document templates
- Correcting documentation errors
- Proposing documentation improvements
- Updating engineering references
- Improving repository organization


## Contribution Workflow

Contributors should follow the standard Git workflow:

1. Fork or clone the repository.
2. Create a new feature branch.
3. Make documentation changes.
4. Review the changes.
5. Commit using descriptive commit messages.
6. Submit a Pull Request.
7. Address review comments if required.
8. Merge after approval.

## Engineering Documentation Principles

Engineering documentation is considered an integral component of every engineering project. Contributors are expected to document their work with the same level of care and professionalism applied to design, development, testing, and validation.

The following principles guide all documentation within this repository:

- **Documentation is Engineering** — Technical documentation is a core engineering deliverable, not an administrative task.
- **Document as You Develop** — Documentation should be created alongside engineering work rather than after project completion.
- **Maintain Traceability** — Requirements, design decisions, revisions, and test results should remain traceable throughout the project lifecycle.
- **Prioritize Consistency** — Follow established templates, terminology, numbering systems, and formatting standards to maintain uniformity across the repository.
- **Preserve Knowledge** — Documentation should enable future contributors to understand, maintain, and extend previous work.
- **Strive for Continuous Improvement** — Documentation standards should evolve through constructive feedback, engineering experience, and collaborative review.


## Documentation Standards 

All documentation should:

- Follow the RJIT-IC-EDS documentation standard.
- Use clear and professional language.
- Maintain consistent formatting.
- Include revision history where applicable.
- Follow repository naming conventions.
- Be reviewed before publication


## Naming Conventions

Contributors should follow established naming conventions for:

- Documents
- Templates
- Figures
- Tables
- Engineering projects
- Revision identifiers


## Branching Strategy

The primary branches are:

- main — Stable documentation
- develop — Ongoing development
- feature/* — New features
- fix/* — Documentation fixes
- release/* — Release preparation


## Commit Message Guidelines

Commit messages should clearly describe the change.

Examples:

- Add proposal template
- Update documentation standard
- Fix revision numbering
- Improve README
- Add CAD documentation


## Pull Request Guidelines

Every Pull Request should:

- Focus on a single logical change.
- Be reviewed before merging.
- Include clear descriptions.
- Reference related issues where applicable.


## Documentation Quality Checklist

Before submitting a contribution, verify that:

- [ ] The document follows the RJIT-IC-EDS documentation standard.
- [ ] File and folder names follow repository naming conventions.
- [ ] Grammar, spelling, and formatting have been reviewed.
- [ ] Figures, tables, and references are correctly numbered and labeled.
- [ ] Revision information has been updated where applicable.
- [ ] All internal links and references are valid.
- [ ] The contribution has been reviewed for technical accuracy.


## Code of Conduct

All contributors are expected to follow the repository's Code of Conduct and maintain a respectful, collaborative, and professional environment.
